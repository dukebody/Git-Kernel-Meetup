What is Git?

And what can it do for you

Git is a Decentralized Version Control System (DVCS)

  • Store snapshots of the current file contents of a project and track code changes between them
  • There is technically no central server keeping track of everything
  • Created by Linus Torvalds on 2005 to replace a commercial alternative

Why the "Git" name?

Linus: "I'm an egotistical bastard, and I name all my projects after myself. First 'Linux', now 'Git'"

('git' is British slang for "pig headed, think they are always correct, argumentative")

Backup copy

Keep a backup copy of a previous state of a project

  • Easy rollback when something breaks
  • See how something was done time ago
  • Checkout the snapshot at any point in time
  • Forget about commenting out, just nuke it!

Track how, when, who, why

Every change (commit) includes:

  • Lines of code changed, in multiple files
  • Date and time
  • Author name and email (to take the blame)
  • Human-readable description of the changes (commit message)

More on tracking...

(git grep / git log)

  • Search regexp through the entire history of the repository
  • Search for commits whose changes include your regexp
  • Search for commits from Pepito with the word "paella" in the last month
  • Search for commits affecting a certain file
  • Binary search to identify faulty commit (git bisect)

Branch out your work!

(you can always merge back later)

Create as many extra branches as you want:

  • Main development
  • Bugfix and feature
  • Experiments

Branches are cheap, Git compresses files across snapshots storing only their differences

Collaborate with others

  • Peer / hierarchical review
  • Comment on changes
  • Iterative submissions + accept / reject
  • See who wrote any given line, and when

Collaborating with others

Remotes

  • They are remote repositories
  • Usually only one, origin, pointing to GitHub, BitBucket, etc.
  • Local branches can track remote branches (master â†’ origin/master)

Basic work with remotes (demo)

git clone URL [folder]

Clone all branches of remote repository, adding remote tracking heads for all them.

git pull [remote-name] [remote-branch]

Update remote-branch head and merge it into the current branch.

git push [remote-name] [local-branch]

Send local commits in local-branch to remote-name.

git remote show remote-name

Show extensive info about the remote, its branches, local branches tracking them...

Some advice ...

  • Make sure you have latest version of a branch from upstream before pushing new changes. Pull before commiting.
  • Rule of thumb: never change anything in a remote repository others might be basing their work on

... and some tips!

  • git config [--global] user.email your_email@example.com
  • git config [--global] user.name John Doe
  • Avoid retyping passwords: SSH keys and credentials caching

Git Workflows

What are workflows?

  • Branching structure
  • Merging rules
  • Standard workflows: centralized, feature branch, Gitflow, forking, etc.

The Feature Branch Workflow

Branching structure

  • Main branch (master)
  • Feature 1
  • Feature 2

Merging rules

  • Keep feature branches in sync with master.
  • When feature is ready, request pull to master.

The Gitflow Workflow

Branching structure

Top to bottom:

  • Production
  • Hotfixes
  • Releases
  • Development
  • Features

Merging rules

  • When feature ready, request pull to develop.
  • If enough new features, create release branch
  • No new features in release branch (bugfixes, docs, etc.)
  • If release ready, request pull to production
  • Tag each new commit in production branch
  • If bug found in production branch, create hotfix branch!
  • If hotfix ready, pull to production (sync everything else!)
  • In general, keep lower branches in sync with higher branches.

The Forking Workflow

Branching structure

  • "Official" repository
  • Developer A's repository
  • Developer B's repository

Merging rules

  • Each developer forks official repository, creates local copy.
  • Developers work on features locally (keep branchin'!).
  • Developers push commits to their own repository.
  • Pull requests filed to project maintainer.
  • Maintainer decides whether to integrate request into official repository
  • Developers keep their forks in sync with official repository.

Bottom line

  • Use workflow that suits your needs (team size, project type)
  • Create your own workflows
  • Crucial: define workflow at start of your project and stick to it.
  • Workflow should be clear to all developers.
